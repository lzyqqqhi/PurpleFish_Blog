            # 跨域

            # 一、定义

            跨域指的是客户端脚本试图从与该脚本的 origins (opens new window)（协议、域名、端口）不同的源加载资源时所面临的安全限制。协议、域名、端口只要有一个不同，就属于跨域。

            # 二、产生原因

            1. 同源策略 ：出于安全考虑，浏览器的同源策略限制了从一个源加载的脚本获取或操作另一个源的资源。例如，当一个网页通过 AJAX (opens new window) 请求或其他方式试图访问不同源的资源时，浏览器会阻止该请求。
            2. 浏览器安全机制 ：现代浏览器遵循同源策略来防止恶意脚本获取用户敏感信息。跨域限制的目的是确保用户的隐私和数据安全。

            # 三、常见跨域场景

            1. 不同域名 ：如从域名 A（www.example.com）请求域名 B（www.anotherdomain.com）的资源。
            2. 不同协议 ：从 HTTP 协议的页面请求 HTTPS 协议的资源，或者反之。
            3. 不同端口 ：在同一域名下，但端口不同的两个应用之间进行通信。

            # 四、解决方法

            # 1. CORS（Cross - Origin Resource Sharing，跨域资源共享） - 主流且标准方法 常见请求头 响应头

            • 原理 ：服务器响应时设置特定 HTTP 头，允许指定源的网页访问本资源。

            • 核心 HTTP 响应头 ：

              • Access - Control - Allow - Origin ：允许的域名。
              • Access - Control - Allow - Methods ：允许的方法（GET、POST 等）。
              • Access - Control - Allow - Headers ：允许携带的自定义头。

            • 示例 ：

              Access-Control-Allow-Origin: https://example.com

            • 流程 ：

              • 浏览器发起跨域请求，服务器若带上正确的 CORS 响应头，浏览器则允许前端访问响应内容，否则拦截（前端拿不到响应数据）。

            # 2. JSONP（仅支持 GET，老旧方式）

            • 原理 ：利用 <script> 标签不受同源策略限制，通过回调函数获取数据。
            • 实现方式 ：
              • 前端动态创建一个 <script> 标签，将需要请求的 URL 设置为该标签的 src 属性。
              • 在 URL 后面添加一个回调函数名的参数，例如 callback=handleResponse。
              • 服务器收到请求后，将数据作为参数，调用前端指定的回调函数，返回一个 JavaScript 脚本。
              • 前端定义的回调函数会自动执行，从而获取到服务器返回的数据。
            • 适用场景 ：仅适合兼容性要求高、只用 GET 请求的场景。

            # 3. 跨域资源嵌入

            • <img>、<link>、<script>、<iframe> 等资源本身可以加载不同源内容，但不能直接访问其内部数据。
            • 利用 postMessage 可实现 iframe 间安全通信。

            # 4. 服务器代理

            • 原理 ：前端请求自己的服务器，由服务器再去请求其他域的数据并转发给前端，实现 “同源”，绕过跨域限制。

            # 五、CORS 的类型

            • 简单请求 ：如 GET、POST（Content - Type 为 application/x-www-form-urlencoded, multipart/form-data, text/plain），会直接请求并由服务器决定是否放行。
            • 预检请求（Preflight） ：对于复杂请求，浏览器会先发一个 OPTIONS 请求检测是否允许跨域，再决定是否发送实际请求。

            # 六、面试标准答案模版

            浏览器为了安全采用同源策略，防止不同源之间的数据随意获取。若需要实现跨域资源共享，主流做法是由服务端设置 Access-Control-Allow-Origin 等 CORS 相关 HTTP 响应头进行授权，浏览器检测到这些头部后允许前端获取响应数据。此外，还有 JSONP（仅 GET），iframe + postMessage、服务器代理等方案，但最推荐且最标准的是 CORS。

            # 注意事项

            • CORS 必须服务端支持并设置相应头部，前端无法强制越权获得其他域数据
            • 预检请求需妥善处理（尤其是企业级 API 或带 token 情况下）
            • CORS 还可控制允许的请求头、方法、是否携带凭证（如 cookie）
