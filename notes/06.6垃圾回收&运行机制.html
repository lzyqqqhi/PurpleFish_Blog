# 垃圾回收 & 运行机制

  • 垃圾回收（Garbage Collection, GC）：自动检测不再被引用的对象，并释放其内存空间。
  • JavaScript 是“具有自动垃圾回收机制”的语言，开发者无需手动释放绝大多数内存。

# 引用计数 js高级程序设计

  • 机制：每个对象有一个引用计数，被引用时 +1，被解除引用时 -1。引用计数为 0 时可回收。
  • 缺点：无法处理“循环引用”问题。

# 标记-清除 (Mark and Sweep)

  • 主流机制，现代浏览器通用
  • 流程：
    • GC Roots（全局对象、当前执行上下文的变量）出发，遍历所有可以到达的对象，做上“被标记”。
    • 未被标记的对象视为不可达（即“垃圾”），统一清除释放。
  • 优势：能有效解决循环引用问题。

# 3. 标记-整理 & 分代回收 回收

  • V8 等引擎优化，通过新生代（对象存活时间短）、老生代（存活时间长）区分回收策略
  • 新生代：采用Scavenge 算法（复制+标记清除，速度快，少量对象） Scavenge
  • 老生代：采用标记-清除&标记-整理（腾出连续空间，防止内存碎片）

# 新生代回收 —— Scavenge 算法

  • 使用 复制算法（Copying GC）
  • 将内存分为两个半区：From Space & To Space
  • 步骤：
    • 活动对象从 From 复制到 To
    • 清空 From
    • 交换两者角色

优点：

  • 快速
  • 避免内存碎片

缺点：

  • 需要双倍内存
  • 不适合大对象

# 老生代回收 —— Mark-Sweep & Mark-Compact

# Mark-Sweep（标记-清除）
  • 标记所有存活对象
  • 清除未标记对象
  • 有内存碎片问题
# Mark-Compact（标记-整理）
  • 在 Sweep 后整理空间
  • 将存活对象向一侧移动，消除碎片
# 垃圾回收技术
  • 并行回收：通过多个辅助线程同时处理垃圾回收任务，加速执行速度，减少主线程停顿时间 。
  • 增量标记：将垃圾回收任务拆分成多个小片段，间歇性执行，避免长时间主线程停顿（2011 年 V8 从全停顿标记切换到增量标记） 。
  • 并发回收：允许辅助线程在后台完成垃圾回收操作，主线程可继续执行 JavaScript 代码，避免挂起 。
  • 惰性清理（Lazy Sweeping）：在增量标记后，延迟清理非活动对象内存，按需逐步清理，减少主线程停顿 。

# 具体触发时机（按代划分）：

# 1. 新生代（Young Generation）
  • 触发条件：当新生代空间（通常是 Semi-Space，默认约 1~16MB）快被填满时。
  • 回收方式：Scavenge（快速复制存活对象到老生代或清理掉）。
  • 特点：频繁触发、停顿时间短（通常 <1ms）。

# 2. 老生代（Old Generation）

  • 触发条件：
    • 老生代空间占用达到阈值（动态调整，默认约 几百MB）。
    • 新生代对象晋升到老生代，导致老生代空间不足。
    • 系统内存压力（如浏览器整体内存紧张）。
  • 回收方式：Mark-Sweep + Mark-Compact（标记-清除 + 整理）。
  • 特点：触发频率低，但停顿时间可能较长（V8 通过增量/并发优化降低影响）。

# 3. 特殊情况

  • 显式触发：开发者可通过 global.gc()（需启动 Node.js 时加 --expose-gc）手动触发 GC，但不推荐。
  • 空闲时 GC：V8 会利用浏览器/Node.js 事件循环的空闲时间（Idle GC）主动回收，减少用户感知延迟。

# 新生代晋升老生代

  • 新生代对象经历一次 GC 后仍存活
  • To 空间使用率超过 25%
  • 大对象直接进入老生代

# 如何优化内存与避免泄漏

  • 尽量减少全局变量
  • 用完及时解绑事件、清空定时器
  • 控制闭包持有范围
  • 较大的数据及时置为 null，断开引用
